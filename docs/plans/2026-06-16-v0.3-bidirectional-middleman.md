# callsign v0.3 — Bidirectional iMessage Middle-Man Operator

**Date:** 2026-06-16
**Author:** Argus (Claude Code, Opus 4.7 1M)
**Status:** Draft — pending GPT-5 + DeepSeek adversarial review

## Problem (from v0.2 audit, 2026-06-16)

v0.2 shipped:
- ✓ `claim`, `list`, `lookup`, `retire`, `route`, `banner`, `send`, `names`
- ✓ SessionStart hook scaffold
- ✓ `imsg-callsign` outbound wrapper (`<Name>: <body>`)
- ✓ `callsign router` parser

v0.2 did NOT ship:
- ✗ Auto-claim on session start — agent must manually run `callsign claim <Name>`; if it forgets, the session never registers (confirmed: this exact session was unregistered until manually fixed)
- ✗ `callsign-router-daemon` under launchd — script exists but no plist, never runs
- ✗ Dispatch from `callsign router` into the matched Claude session — current router only LOGS the routing decision
- ✗ Reply capture from Claude session back into iMessage — no pipe exists
- ✗ SessionStart hook's Python heredoc fails silently mid-script — `additionalContext` never delivered

Result: callsign claims a name, but iMessages addressed to that name go nowhere and no reply ever returns.

## Goal

callsign acts as a true bidirectional middle-man:

```
iMessage in ──► router daemon ──► dispatch to matched session ──► reply ──► iMessage out
              (launchd)         (claude --session-id <UID>)     (capture)  (imsg send)
```

Every Claude Code session auto-claims a name on start (bound to its `$CLAUDE_SESSION_ID`), so routing has a deterministic target.

## Architecture

```
                       ┌───────────────────────────────────────┐
                       │     ~/.callsign/registry.db (sqlite)  │
                       │  callsign│session_uid│project│pid│... │
                       └──────────┬───────────────────────────┬┘
                                  │                            │
   ┌───────────────────┐          │                            │
   │ Claude Code start │──claim──▶│                            │
   │  (SessionStart)   │          │                            │
   └───────────────────┘          │                            │
                                  ▼                            │
   ┌──────────────┐  imsg     ┌─────────────────┐  match       │
   │ iMessage in  │──watch──▶│  router daemon   │──lookup────▶ │
   └──────────────┘  --json   │  (launchd)       │              │

                                       │ dispatch               │
```

## Components to build

### 1. Auto-claim on SessionStart (fixes blank-session bug)

Rewrite `~/.claude/hooks/callsign_session_start.sh` so that:

- If the session is already in registry → reuse the existing callsign
- If not → call `callsign claim --auto` which:
  - Picks an unused name from `names.py` pool
  - Inserts row with `(callsign, session_uid=$CLAUDE_SESSION_ID, project_path=$CLAUDE_PROJECT_DIR, pid=$$, claimed_via='auto', claimed_at=NOW, status='active')`
  - Returns the chosen name
- Hook then outputs `additionalContext` JSON:
  ```json
  { "additionalContext": "Your callsign for this session is **<Name>**. Outgoing iMessages: `imsg-callsign \"<body>\"`. You may override with `callsign claim <NewName>` if you prefer a different name." }
  ```

**Fix the Python heredoc bug** — current hook uses `python3 - "$ARG1" "$ARG2" <<'PY' ... PY` which silently fails partway through. Refactor to a standalone Python script invoked by the bash hook with args, so errors surface to stderr and don't break the JSON output.

### 2. Router daemon (launchd-managed)

Daemon body (already shipped — verify it still works):
```bash
while true; do
    imsg watch --json 2>>"${LOG_DIR}/router.stderr" \
        | callsign router --dispatch \
        >>"${LOG_DIR}/router.stdout" 2>>"${LOG_DIR}/router.stderr" || true
```

### 3. Dispatch (the missing middle)

Add `--dispatch` flag to `callsign router`:

```python
# src/callsign/router.py — add to existing route()
if args.dispatch and hit:
    inbox_append(hit.callsign, msg)              # ~/.callsign/inbox/<Name>.jsonl
    if not quiet_hours_now():
        dispatcher.fire(hit.session, hit.body, msg.sender)
```

`src/callsign/dispatcher.py` (new):
```python
def fire(session: Session, body: str, sender: str) -> None:
    """Run one Claude turn against session.session_uid; pipe reply to imsg."""
    with locks.acquire(session.session_uid, timeout=120):
        if processed_db.seen(msg.guid):
            return  # idempotency
        proc = subprocess.run(
            ["claude", "--session-id", session.session_uid, "--print", body],
            capture_output=True, text=True, timeout=600,
        )
        reply = clean(proc.stdout)
        for chunk in chunked(reply, max_chars=3500):
            subprocess.run(
                ["imsg-callsign", "--to", sender, chunk],
                env={**os.environ, "CALLSIGN": session.callsign},
                check=False,
            )
        processed_db.mark(msg.guid)
```

### 4. Idempotency

`~/.callsign/processed.db` (sqlite) — single table:
```sql
CREATE TABLE processed (
    guid TEXT PRIMARY KEY,
    callsign TEXT NOT NULL,
    received_at REAL NOT NULL,
    dispatched_at REAL,
    reply_sent_at REAL
);
```

Daemon checks `processed.guid` before firing dispatcher. Survives daemon restart.

### 6. Concurrency

- Bounded semaphore (4 concurrent dispatchers) in the daemon
- Per-session-UID lock file `~/.callsign/locks/<UID>.lock` (flock) so concurrent messages to the same callsign serialize
- If lock contended >120s, send "busy, queued" via iMessage and stash in inbox

### 7. Quiet hours

Per daniel's global rule: NO iMessage to him 23:00-06:00 Central. Apply same to inbound dispatch:
- During quiet hours, write to inbox but do not dispatch
- At 06:00 next morning, drain queued inbox in FIFO order

### 8. Reply chunking

iMessage practical limit ~16KB; some carriers fragment at 4KB. Chunk at 3500 chars on sentence boundaries, send each as a separate iMessage. Insert `[1/N]`, `[2/N]` markers if N > 1.

### 10. Failure modes

| Failure | Behavior |
|---|---|
| `claude --session-id <UID>` exits non-zero | Mark callsign retired; reply "session offline" |
| Session UID not found in registry | Fall back to lead session (last claimed for project) |
| `imsg send` fails 3x | Log + dump reply to `~/.callsign/dead-letter/` + alert daniel |
| `imsg watch` stream dies | launchd restarts via KeepAlive; resume from last processed GUID |
| Dispatch timeout (600s) | Kill child, send "timed out" reply, mark guid processed |
| Quiet hours arrival | Queue in inbox; drain at end-of-quiet |

### 13. Open risks (review targets)

1. **`imsg watch --json` flag:** v0.11.0 of `imsg` lists `watch` but not `--json` explicitly. Need to verify the flag actually emits structured JSON. If not, parse plain output or patch imsg.
2. **`claude --session-id <UID> --print "<body>"`:** does Claude Code CLI support headless one-shot invocation against a specific session UID? Need to verify before committing. Fallback: write body to `~/.callsign/inbox/<Name>.jsonl` and rely on next-user-prompt hook injection (slower, but works).
3. **Re-entrancy:** if my current foreground session is mid-task and an iMessage triggers `claude --session-id <my-UID>`, two `claude` processes may attempt to mutate the same session state. Per-session lock mitigates but doesn't eliminate. May need to dispatch ONLY when no active foreground process holds the session (PID check).
4. **State race on `processed.db`:** if router daemon and a manual `callsign router --dispatch` both run, SQLite WAL needed (`PRAGMA journal_mode=WAL`).
5. **Reply chunk reordering:** chunked iMessages may arrive out of order. Use `[k/N]` prefixes so daniel can resequence.
6. **GUID source:** does `imsg watch --json` expose Apple's `guid`? If not, idempotency key must be `(sender, ts, hash(text))` — collision-resistant enough for our volume.
7. **Long replies + Claude Code streaming:** `claude --print` outputs the final message only? Or streams partial tokens? If streaming, we need to wait for done-event before chunking + sending.
8. **`claim --auto` on parallel session starts:** if daniel opens 3 Claude Code windows simultaneously, all 3 may race to claim the same name from the pool. Need a SQLite `INSERT ... ON CONFLICT` or transactional lock on the names table.

## Acceptance criteria

1. Open a fresh Claude Code window → SessionStart hook auto-claims a name → `callsign list` shows it with the session UID.
2. iMessage `Argus, what time is it?` from daniel's phone → reply lands in the same iMessage thread, prefixed `Argus: ...`, within 60 seconds.
3. iMessage at 02:00 → no reply until 06:00 next morning; queued reply then arrives.
4. Two messages back-to-back to Argus → both processed in order, no duplicate replies.
5. Kill `claude --session-id` mid-dispatch → daemon recovers, sends timeout reply, doesn't lose other in-flight dispatches.
6. `callsign status` shows daemon up, recent messages processed, zero errors.
